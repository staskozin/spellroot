# AGENTS.md

## О проекте

Spellroot — мод для Minecraft на NeoForge. Проект находится на ранней стадии. Текущий игровой контент ограничен предметом Ender Focus, его клиентскими ресурсами, локализациями и рецептом крафта.

Основные параметры проекта:

- Minecraft: `26.2`.
- NeoForge: `26.2.0.18-beta`.
- Java toolchain: `25`.
- Gradle wrapper: `9.2.1`.
- Mod ID и namespace ресурсов: `spellroot`.
- Базовый Java package: `ru.staskozin.spellroot`.

## Структура репозитория

- `src/main/java/ru/staskozin/spellroot/Spellroot.java` — общая точка входа мода.
- `src/main/java/ru/staskozin/spellroot/registry/ModItems.java` — Deferred Register предметов и добавление предметов в стандартные вкладки творческого режима.
- `src/main/resources/assets/spellroot/` — клиентские ресурсы мода: локализации, модели и текстуры.
- `src/main/resources/data/spellroot/` — серверные data pack-ресурсы мода: рецепты и связанные advancement.
- `src/main/templates/META-INF/neoforge.mods.toml.template` — шаблон метаданных мода. Значения `${...}` подставляются из `gradle.properties` задачей `generateModMetadata`, а результат получает имя `neoforge.mods.toml`; не редактируйте сгенерированную копию в `build/`.
- `src/generated/resources/` — ожидаемый каталог результатов data generation; он подключен к main resources.
- `build.gradle` — конфигурация NeoForge ModDev, dev-запуски `client`, `server`, `gameTestServer` и `data`, сборка и публикация.
- `.github/workflows/build.yml` — CI на JDK 25, запускающий Gradle `build` для push и pull request.

Каталоги `.gradle/`, `build/` и `run/` являются локальными/сгенерированными и не должны использоваться как источник истины или попадать в коммиты.

## Главное требование: multiplayer-first

Spellroot предназначен для мультиплеера. Любую игровую фичу проектируйте и реализуйте так, чтобы она корректно работала на dedicated server с несколькими одновременно подключенными игроками, а не только в одиночной игре или на integrated server.

- Сервер является единственным авторитетным источником игрового состояния. Клиент может отправлять намерение игрока и показывать предсказание/эффекты, но не должен сам подтверждать урон, расход ресурсов, cooldown, инвентарь, прогресс, положение или результат заклинания.
- В обработчиках входящих payload/packet всегда повторно проверяйте на сервере отправителя, его право на действие, дистанцию, измерение, актуальное состояние, доступность цели и необходимые ресурсы. Не доверяйте entity ID, координатам, числам и другим значениям от клиента; учитывайте спам и повторную доставку действий.
- Изменяйте мир и общие данные только на логической стороне сервера. Синхронизируйте клиентам минимально необходимое состояние стандартными механизмами NeoForge/Minecraft или явно определенными payload. Не рассчитывайте, что `static`-поля или память клиента и сервера общие.
- Не импортируйте `net.minecraft.client.*` и другие client-only классы из common/server-кода. Рендеринг, экраны, ввод, частицы и локальные звуки размещайте в client-only классах с корректным `Dist.CLIENT`; классы, загружаемые dedicated server, должны быть безопасны без клиентских зависимостей.
- Учитывайте late join, disconnect/reconnect, respawn, смену измерения, выгрузку чанков/сущностей и одновременные действия нескольких игроков. Новому наблюдателю должно приходить актуальное состояние, а данные одного игрока не должны утекать другому без необходимости.
- Не храните изменяемое состояние конкретного мира, уровня, сущности или игрока в глобальных singleton/static-полях. Выбирайте хранилище с подходящим жизненным циклом и явно задавайте сохранение и синхронизацию.
- Клиентские визуальные эффекты запускайте на основании подтвержденного сервером события либо делайте безопасное предсказание с возможностью согласования. Избегайте двойного исполнения логики на обеих сторонах.
- Конфигурацию с правилами геймплея трактуйте как серверную/общую; клиентские настройки не должны давать игрового преимущества или менять серверный результат.

## Правила разработки

- Используйте Deferred Register для реестров NeoForge. Идентификаторы, translation keys и пути ресурсов должны оставаться в namespace `spellroot` и быть в `snake_case`.
- По мере появления реальных механик выносите реестры, networking, серверную игровую логику и клиентское представление в отдельные классы/пакеты; не наращивайте монолитный `Spellroot.java`.
- Подписывайте обработчик на правильную шину событий: lifecycle/registry-события — на mod event bus, игровые события — на `NeoForge.EVENT_BUS`.
- Не выполняйте потенциально небезопасную для потока работу прямо в сетевом callback; переводите изменение игрового состояния на игровой поток средствами API.
- Для пользовательского текста добавляйте translation keys вместо захардкоженных строк. Поддерживайте как минимум существующий `en_us.json`; при появлении русской локализации используйте отдельный `ru_ru.json`.
- Не редактируйте артефакты в `build/`. Источниками метаданных являются `gradle.properties` и `src/main/templates/META-INF/neoforge.mods.toml.template`.
- Сохраняйте UTF-8 и существующий стиль Java. Удаляйте неиспользуемые импорты и шаблонный код, когда он перестает быть нужен.
- Не меняйте версии Minecraft, NeoForge, Java или Gradle как побочный эффект фичи: такие обновления должны быть отдельным осознанным изменением.

## Сборка и запуск

На Windows используйте `gradlew.bat`, на Linux/macOS — `./gradlew`.

- `./gradlew build` — обязательная базовая проверка: компиляция, тесты и сборка JAR в `build/libs/`.
- `./gradlew runClient` — запуск dev-клиента.
- `./gradlew runServer` — запуск dedicated dev-сервера без GUI.
- `./gradlew runGameTestServer` — запуск GameTest server. Сейчас тестов в проекте нет, и конфигурация может завершиться ошибкой при отсутствии зарегистрированных GameTest.
- `./gradlew runData` — генерация ресурсов в `src/generated/resources/`.
- `./gradlew --refresh-dependencies` — использовать только при проблемах с кешем зависимостей; `./gradlew clean` — при необходимости полной пересборки.

Перед завершением изменения как минимум запустите `build`. Для игровой или сетевой механики одной компиляции недостаточно: проверьте dedicated server и два клиента, если среда это позволяет. Минимальные сценарии — оба игрока видят одинаковый результат, повторное подключение/late join восстанавливает состояние, смена измерения и respawn не ломают данные, а некорректный или повторный запрос клиента безопасно отклоняется. Если ручная multiplayer-проверка не выполнена, явно укажите это в отчете.

## Тесты и критерии готовности

Автоматических тестов и GameTest в репозитории пока нет. Для новой нетривиальной серверной механики по возможности добавляйте GameTest, особенно для изменений мира, сохранения состояния, прав доступа и взаимодействия нескольких сущностей/игроков.

Изменение считается готовым, когда:

- проект успешно проходит `build` на JDK 25;
- common-код загружается на dedicated server без client-only классов;
- сервер валидирует и авторитетно применяет все влияющие на игру действия;
- состояние корректно сохраняется и синхронизируется для нескольких игроков, включая late join/reconnect, где это применимо;
- добавлены необходимые ресурсы и локализации;
- в коммит не включены локальные данные из `run/`, кеши или результаты `build/`.
