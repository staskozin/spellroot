Сгенерируй сообщение для git-коммита.

Если пользователь передал описание — используй его.

Если описание не передано, сначала выполни `git diff --cached` и сгенерируй сообщение на основе staged changes. Если индекс пуст, выполни `git diff` и учти незастейдженные изменения. Сообщай, что данных недостаточно, только если нет ни описания, ни staged, ни незастейдженных изменений.

## Правила

1. Используй формат Conventional Commits: `type(scope): описание`.
2. Пиши сообщение на русском языке.
3. Начинай описание со строчной буквы, без точки в конце, с глагола прошедшего времени: например, «добавлен», «исправлен», «обновлён», «удалён», «реализована».
4. Scope необязателен, но желателен: используй имя компонента, файла или области, например `components`, `UiButton`, `package.json`, `storybook`.
5. Выводи каждое предложенное сообщение в отдельном многострочном моноширинном блоке для удобного копирования.

## Типы

- **feat** — новая функциональность (minor release).
- **fix** — исправление бага (patch release).
- **build** — изменения сборки или зависимостей.
- **chore** — рутинные задачи, не затрагивающие исходный код и тесты.
- **ci** — изменения CI/CD.
- **docs** — изменения документации.
- **perf** — оптимизация производительности.
- **refactor** — рефакторинг, не являющийся `fix` или `feat`.
- **revert** — откат коммита.
- **style** — форматирование или линтинг, но не CSS-изменения.
- **test** — добавление или исправление тестов.

## Breaking changes

Если изменение ломает обратную совместимость, добавь `!` после типа или scope: `feat!: редизайн API UiButton`.

## Действия

1. Проанализируй описание пользователя, staged changes или незастейдженные изменения — в таком порядке приоритета.
2. Определи подходящий тип и scope.
3. Сформулируй краткое описание.
4. Выведи готовое сообщение коммита.
5. Если описание неоднозначно, предложи 2–3 варианта.
